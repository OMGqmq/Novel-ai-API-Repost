## 🛠️ 重构设计与决策历程

### 1. 现状分析
我们分析了项目中的原生交互，发现了以下影响质感的问题：
* **原生 Prompt**：卡密（VIP Key）输入和管理员密码解锁功能此前使用的是浏览器原生的 `prompt()` 弹窗，体验极其简陋且平台表现不一致。
* **图库切换割裂**：在图库中点击缩略图会直接将整个右侧试图强行切回“主画布”并在大图下展示操作面板，导致在图库中无法连续浏览和对比参数。
* **API Key 输入体验差**：原本的多 API Key 输入方式是在一个 `textarea` 里让用户手动换行，易出错且排版简陋。

### 2. 确立的优化方案
经过我们的沟通（`grill-me` 互动设计过程），达成了以下重构决策：
1. **Glassmorphism 模态框系统**：在 HTML 中植入统一的高级毛玻璃和渐变微发光弹窗，封装统一的 JS 打开/关闭淡入缩放动画，彻底替换掉所有的原生 `prompt()`。
2. **多 Key 动态输入行**：将 API Key 配置重构为动态输入行。默认显示一个输入框，用户可以点击 `+` 增加 Key 输入框，或者点击“垃圾桶”删除行。
3. **元数据升级**：升级了 `GalleryStore` ( IndexedDB ) 存储格式，在生成图片成功时自动生成 32 位无符号随机 seed，并将全量生成参数（CFG, Steps, Sampler 等）存入 meta 中。
4. **全功能大图详情灯箱 (Lightbox)**：在图库中点击缩略图时直接拉起灯箱。
   * **翻页**：桌面端支持悬浮箭头与键盘左右方向键翻页；移动端支持滑动手势（Swipe）切换。
   * **面板**：桌面端双栏展示大图与元数据，移动端通过可拖拽的 Bottom Sheet 展开/折叠。
   * **快捷创作**：提供“套用参数回填主面板”、“一键送入局部重绘”和“送入无限扩图”等打通工作流的操作。

## 🔧 后续缺陷修复与性能调优记录

### 1. 缺陷排查 (Bug Fix)
* **详情面板遮挡 (桌面/移动端)**：
  * **问题**：在打开 Lightbox 时只能看见大图，侧边栏详情和功能按钮缺失。原因是侧边栏的移出过渡属性优先级过高，在灯箱激活态下未被拉回屏幕。
  * **修复**：修改 CSS，限制桌面端 `.modal-active #lightboxSidebar` 在最小宽度 `768px` 下拉回至 `translate(0, 0)`。
* **移动端参数抽屉失效 (Mobile)**：
  * **问题**：手机端在点击顶部把手时抽屉打不开。原因是被 flex 文档流的大图完全挤到屏幕下方，且平移属性受桌面端全局 `!important` 覆盖锁死。
  * **修复**：在手机端把 `#lightboxSidebar` 升级为绝对定位（脱离文档流），默认为仅露出 50px 拖拽把手的隐藏式抽屉，且点击把手可展开滑入 70% 屏幕高度的 Bottom Sheet，解决由于文档流排挤及权重引起的展示 Bug。

### 2. 核心性能优化 (Performance Tuning)
* **数据读取阻滞卡死问题**：
  * **痛点**：当用户生成的历史图片累积较多（几百张甚至上千张）时，由于单张大图的 Base64 数据量庞大，原本在每次加载和翻页时调用 `store.getAllImages()`（即读取全表反序列化）会造成严重的 CPU/内存阻塞，产生多秒至数十秒的卡死。
  * **优化方案**：
    1. **逆序游标分页 (IndexedDB Cursor Pagination)**：在 `storage.js` 新增了 `getImagesPage(page, pageSize)` 方法。利用 IndexedDB 的 `prev` 游标逆向读取，并通过 `cursor.advance()` 极速跳过条目，每次只反序列化当前分页（如 24 张）的数据送入内存。
    2. **图库无限滚动 (Infinite Scroll)**：在图库主容器上监听滚动事件，当滚动到接近底部时，自动流式异步追加加载下一页图片，页面开启速度提升数万倍（达到毫秒级瞬开）。
    3. **灯箱翻页零读取**：打开 Lightbox 预览大图时，直接复用当前已在内存中加载的 `galleryItems` 列表，拒绝在翻页时二次读库；在大图翻页到已加载边缘时，后台静默自动拉取下一页，实现翻页无卡顿且不间断。
